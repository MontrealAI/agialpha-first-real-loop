# CRA / NIS2 / GDPR Mapping

## Cyber Resilience Act posture

If SecureRails is placed on the EU market as a software product with digital elements, apply secure-development and vulnerability-handling controls:

- secure-by-design development;
- vulnerability disclosure process;
- security update policy;
- user instructions;
- SBOM or dependency inventory where appropriate;
- incident reporting process if applicable.

## NIS2 posture

Customers in essential or important sectors may impose vendor cybersecurity requirements. SecureRails should support them through:

- evidence logs;
- incident runbooks;
- access control review;
- supply-chain metadata;
- audit-ready security records;
- secure deployment documentation.

## GDPR posture

SecureRails should avoid personal data and solely automated significant decisions. If personal data is processed:

- document lawful basis;
- minimize data;
- redact secrets and personal data where feasible;
- complete DPIA where required;
- sign DPA;
- define retention and deletion.

## Claim footer

No Evidence Docket, no empirical SOTA claim. Autonomous evidence production is allowed; autonomous claim promotion is not.
