# Security Policy

## Scope
This policy applies to SecureRails documentation, workflows, scripts, generated artifacts, and associated governance assets in this repository.

## Preferred private reporting
Please use GitHub private vulnerability reporting for this repository when available. Private vulnerability reporting should be enabled by repository administrators before public production deployment.

Contact placeholder (not for production): `security-contact-pending@example.invalid`

Please do not report vulnerabilities through public issues if the report contains sensitive details.

## What to include
- Affected file, workflow, script, or process
- Reproduction steps and impact
- Safe proof-of-concept details
- Suggested remediation

## What not to include
- Raw secrets or credentials
- Malware payloads
- Exploit execution beyond safe proof-of-concept discussion

## Safe harbor
Good-faith research in scope and aligned with this policy will be treated as authorized for coordinated defensive disclosure.

## Out-of-scope activities
- denial-of-service tests
- social engineering
- physical attacks
- spam
- credential stuffing
- attacks against third-party services
- external target scanning
- accessing, modifying, or exfiltrating data
- printing secrets
- malware

## Response expectations
We triage reports, prioritize by severity, coordinate with maintainers, and provide updates as remediation progresses.

## Coordinated disclosure
We follow coordinated vulnerability disclosure principles with private triage first and publication after mitigations are ready.

No bounty is promised unless separately approved in writing.

## Claim boundary
SecureRails reports and Evidence Dockets are advisory governance artifacts. They do not certify security.

SecureRails is AI-agent security governance and proof-bound defensive remediation. It is not autonomous cybercertification claim, not offensive cyber, not a high-risk decision system by intended purpose, not a GPAI model provider by default, and not an investment product.
